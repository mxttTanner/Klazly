# Klazly Parent Portal — Full Audit (2026-07-02)

Audited by 5 parallel reviewers across: multi-tenancy/authorization, auth/session, correctness/data-integrity, RLS/schema, and ops/config/deps. Read-only; no code changed. Findings deduplicated and severity-reconciled below. `file:line` references are clickable.

---

## CRITICAL

### C1 — RLS lets any center admin read/write EVERY center's lessons & student notes; any teacher can write lessons into any center
**Files:** `db/fix-rls-recursion.sql:97-158` (policies `lessons_select`, `lessons_teacher_write`, `slu_select`, `slu_teacher_write`)

When the RLS recursion bug was fixed, the teacher/parent branches got SECURITY DEFINER helpers that preserve scoping (`is_teacher_of_class` → `teacher_id = auth.uid()`; `is_parent_of_student` → the parent link — both safe). But the **admin branch was reduced to a bare `current_user_role() = 'admin'` with no center check**, and — unlike `classes_select`/`students_select` — these four policies have **no outer `center_id = current_user_center_id()` wrapper**. (`lessons`/`student_lesson_updates` have no `center_id` column; they reach it via `class_id → classes`, which is exactly the join the fix removed for admins.)

`lessons_teacher_write`'s `WITH CHECK` is only `teacher_id = auth.uid() or admin`, so a teacher can **INSERT** a lesson row pointing at *any* class in *any* center.

**Why it's live and exploitable:** `NEXT_PUBLIC_SUPABASE_URL` + anon key ship in the client bundle by necessity. Any authenticated user can bypass the app entirely and call PostgREST directly with their own JWT:
```
GET https://<proj>.supabase.co/rest/v1/lessons?select=*
GET https://<proj>.supabase.co/rest/v1/student_lesson_updates?select=*
```
An admin gets every tenant's lessons and per-student `individual_note` (child-level PII); a teacher can forge lessons into other centers' classes. The app's server actions do manual center checks, so the UI doesn't leak it — but **RLS is the security boundary**, and it violates the project's own rule ("RLS on every table, every query scoped to center_id"). *(RLS reviewer found it; authorization reviewer missed it — I read the policies directly to confirm.)*

**Fix:** restore center scoping on all four admin/teacher branches via a center-aware SECURITY DEFINER helper. SQL in the "Fix pack" section.

## HIGH

### H1 — Unauthenticated phone→email PII harvesting via `resolveLoginEmail`
**File:** `src/app/(auth)/login/actions.ts:28-64` (`return { email: hit.email }` at :62)

Exported `"use server"` action, no auth gate, called pre-login. Given a phone number it uses the **service-role client** (RLS-bypassing) to look up `public.users` across all centers and returns the account's **real email** to the browser. The synthetic-email fallback hides *phone-only* accounts, but any parent who registered with a real email has it handed back. Phones are semi-public in VN → scriptable harvest of guardians' emails + phone↔account confirmation, unauthenticated, unthrottled.
**Fix:** resolve phone→email and run `signInWithPassword` in a single server action that returns only success/failure; never echo another user's email to an unauthenticated caller.

### H2 — Paused / past_due / pending_renewal centers are never locked out
**Files:** `src/lib/auth.ts:91-98` vs `src/app/super-admin/actions.ts:1073-1107` (`pauseCenter`), `db/pause-and-cancel.sql`

`requireUser()` only redirects `expired` and `canceled+grace-elapsed` to `/locked`. But `pauseCenter`, the migration comment, and the `/locked` page copy ("Your center is paused") all assume pausing cuts access. It doesn't — a paused non-paying center keeps full read/write indefinitely, via pages *and* direct server-action calls. The operator's only billing-enforcement lever is a no-op.
**Fix:** centralize the lock predicate in `requireUser` and include `paused`, `past_due`, `pending_renewal` per policy.

### H3 — Trial expiry only fires when the super-admin opens the dashboard
**Files:** `src/lib/subscription.ts:223` (`expireOverdueTrials`), only call site `src/app/super-admin/page.tsx:266`; enforcement `src/lib/auth.ts:91`

`requireUser` locks on the *stored* status; the only thing that flips `trial → expired` (or auto-converts founding trials) is a lazy pass on the super-admin dashboard render. No cron, no `trial_ends_at` check in `requireUser`. Don't open the dashboard for two weeks → the center gets two free weeks; founding centers' paid clock never starts.
**Fix:** in `requireUser`, treat `status==='trial' && trial_ends_at < now()` as expired (mirror `deriveStatus`), or add a Vercel cron.

### H4 — Lesson per-student updates accept arbitrary student IDs (no roster/center check)
**File:** `src/app/teacher/classes/[id]/lessons/new/actions.ts:169-190` (`buildUpdates`), used :288-299 / :435-473

`student_id`s come from hidden form fields, validated only as UUIDs. Neither the action nor `slu_teacher_write` RLS checks the student is enrolled in that class or even in the same center (FK only requires existence). A crafted POST — or a stale form after a student is moved — writes behavior/attendance/notes for students not in the class, including other centers', corrupting stats.
**Fix:** fetch the class roster server-side and intersect submitted `student_id`s against it before insert.

### H5 — `ptm_update_read` RLS policy allows rewriting message bodies
**File:** `db/messages.sql` (the `ptm_update_read` UPDATE policy)

Meant for "mark as read" but has no `WITH CHECK` and no column restriction. Via direct PostgREST, any parent/teacher/admin can UPDATE `body` / `sender_user_id` / `read_at` of **any message they can see**, including the other party's. The message log — the exact thing meant to replace Zalo groups as a dispute record — isn't tamper-proof.
**Fix:** move read-marking to a SECURITY DEFINER function that only touches `read_at` and drop the broad UPDATE policy (Postgres RLS can't restrict columns directly).

### H6 — Next.js 14.2.35 past its patch window (RSC DoS advisories)
**File:** `package.json:36`

14.2.35 is the last 14.x; no further backports. Applicable highs: RSC HTTP-deserialization DoS (GHSA-h25m-26qc-wcjf) and two Server-Components DoS (GHSA-q4gf-8mx6-v5v3, GHSA-8h8q-6873-q5fj). **Not** vulnerable to the middleware-auth-bypass CVE-2025-29927 (fixed in 14.2.25; also structurally moot here since authz is in layouts+RLS, not middleware). Real residual risk is the RSC DoS trio.
**Fix:** plan upgrade to Next 15.5.16+ (brings React 19). Biggest single ops item.

## MEDIUM

- **M1 — No rate limiting anywhere** (`login-form.tsx:49`, `login/actions.ts`, `forgot-password-form.tsx:39`). No app-level throttle; relies on Supabase defaults. Credential stuffing, reset-email bombing, and the H1 oracle are all unthrottled. → Vercel WAF rate rules or Upstash Ratelimit on login/resolve/forgot-password.
- **M2 — CSV temp passwords never force-reset** (`admin/import/actions.ts:19-21, 213-222`). Entropy is fine (~72 bits, `crypto.randomBytes`), but "parent expected to reset" is enforced nowhere; the temp password is a permanent standing credential delivered out-of-band (Zalo/paper). → add `must_change_password` flag, force reset on first login.
- **M3 — CSV parent import: timeout mid-loop strands parents** (`admin/import/actions.ts:95-225`). Sequential createUser+insert per row, no cap/batching; if the function times out, users are created but the one-time generated passwords are returned to no one, and re-upload skips them as "existing." → cap rows (~200) or chunk; pair with M2's reset flow.
- **M4 — Student CSV import not idempotent** (`admin/import/actions.ts:281-335`; no unique constraint in `db/schema.sql`). Unlike the parents importer, zero dup detection — re-uploading a corrected roster duplicates every student, splitting attendance/report data. → skip existing `(center_id, full_name, class_id)`.
- **M5 — Worksheets & logos buckets are world-readable by URL** (`db/worksheets.sql:9-12`, `db/storage.sql:11-14`). `public=true` + SELECT policy `using (bucket_id='worksheets')` — path is namespaced by center but that's obscurity, not access control. Worksheets may contain student-identifiable material / answer keys. → private bucket + short-lived signed URLs, or path-prefix RLS. (Logos are fine public.)
- **M6 — Lazy/manual subscription lifecycle leaves rows un-billable.** Founding auto-conversion sets `next_billing_at` but not `subscription_ends_at`, so renewal detection (keyed on `subscription_ends_at`) never nudges them (`subscription.ts:266-279` vs `:173-185`). Manual `updateSubscriptionStatus`/`updateSubscriptionPlan` (`super-admin/actions.ts:456-484, 536-552`) have the same blind spot and don't recompute periods. → funnel activation through `convertCenterToPaid` / stamp both dates.
- **M7 — Month-end billing date overflow** (`super-admin/actions.ts:290-305, 779-815`; `subscription.ts:263-265`). `setMonth(getMonth()+1)` on Jan 31 → Mar 2/3; billing anchor drifts 1-3 days/cycle. → clamp to end-of-month or use a date lib.
- **M8 — No idempotency on lesson create or message send** (`createLesson`; `messages-actions.ts:67-77`). Buttons disable while pending, but network retries / back-button / two tabs still double-submit → duplicate lessons double-count stats, duplicate parent emails. → unique index on `(class_id, lesson_date)` if that's the model, or an idempotency key.
- **M9 — `markThreadRead` runs during RSC render** (`parent/students/[id]/page.tsx:181-185`). Marks all teacher messages read when the parent opens the *lessons* tab (default) → teacher "read" signal lies; and `revalidatePath` during render throws, swallowed by `.catch(()=>{})` (masked only because pages are `force-dynamic`). → only mark on `activeTab==='messages'`, move out of render.
- **M10 — Resend email failures invisible to Sentry** (`src/lib/email.ts:99-105`, `messages-actions.ts:169`). Failures go to `console.error` only; a broken RESEND key = parents silently stop getting notifications. → `Sentry.captureException(err,{tags:{area:'email'}})`.
- **M11 — Fixable transitive vulns** (`npm audit`): `ws` (mem-disclosure + DoS, via supabase-js realtime) and `qs` DoS — non-breaking `npm audit fix`. Also bump `@sentry/nextjs` 10.52.0 → 10.63.0 (OTel baggage DoS, semver-compatible).
- **M12 — 27 loose hand-applied SQL migrations, no ordering/tracking** (`db/`). Drift already papered over at runtime with try/fallback selects (`auth.ts:26-40` for `phone`, `requireUser` for `subscription_ends_at`). → adopt `supabase migration` or a numbered convention + applied-log; delete the runtime fallbacks once prod is confirmed.

## LOW

- **L1 — `submitFeedback` is an unauthenticated service-role write** (`feedback-actions.ts:25-60`) — uses `getCurrentUser()` (nullable) then always inserts. Spam vector into the super-admin inbox. → `requireUser()` or rate-limit.
- **L2 — No Content-Security-Policy header** (`next.config.mjs:13-35`). Good baseline (HSTS preload, nosniff, XFO, Referrer/Permissions-Policy) but no CSP backstop for stored-XSS in user-authored notes/messages. Allowlist Clarity + Sentry; start report-only.
- **L3 — Worksheet delete authz is app-layer only** (`db/worksheets.sql` `worksheets_write` is `for all` to any teacher in the center; app checks `uploaded_by` at `worksheets/actions.ts:102`) — bypassable via direct PostgREST. Same class of issue as C1/H5.
- **L4 — `worksheet_id` on lessons trusted raw** (`.../lessons/new/actions.ts:232-234, 379-383`) — not UUID-validated (non-UUID → raw Postgres error to the teacher), not checked against `worksheets.center_id`. → `z.string().uuid()` + center check.
- **L5 — Reset-password policy client-side only** (`reset-password-form.tsx:41-55`) — 8-char min enforced only client-side; server floor is Supabase default (6); other sessions not explicitly revoked on reset. → set policy in Supabase Auth; enable "revoke other sessions."
- **L6 — Subscription-lifecycle edge cases:** `extendTrial` on a canceled center leaves it locked (`super-admin/actions.ts:618-622`); `reactivateCenter mode=resume_trial` ignores an already-passed `trial_ends_at` (`:1265-1297`); tolerant column-strip regexes (`:89-96, 357-386`) can drop a field on an unrelated error and still report success; `deleteCenterCascade` swallows errors to console and returns void (`:988-1041`, ordering is safe now).
- **L7 — Timezone edges (VN UTC+7):** trials expire at creation time-of-day not end-of-day (`createCenter:273-277`); 30-day parent report window carries time-of-day (`parent/students/[id]/page.tsx:267-272`); parent-home "last lesson" computed from 50 most-recent across all children → a busy class can starve a sibling's (`parent/page.tsx:98-106`); `process.env.TZ` set in `instrumentation.ts` is ignored by Node on Windows dev.
- **L8 — Info-leak oracles:** "phone already used at another center" (`admin/parents/actions.ts:103-108`, `teachers/actions.ts:82-87`) confirms cross-center existence to an admin; super-admin KPI counts include the hidden demo center (`super-admin/page.tsx:227-252`).
- **L9 — Housekeeping:** `shadcn` CLI is a runtime dep (drags `hono`, flagged) → move to devDeps; ESLint 8 EOL (resolves with Next 15); 3 merged local branches (`pitch-audit-fixes`, `site-redesign`, `ui-revamp`) safe to delete; guard scripts use a fragile Windows path hack (`fileURLToPath()` instead) but pass today.

## Verified solid (coverage)
- **Escalation vector closed:** no self-service UPDATE policy on `public.users` → a user cannot change their own `role`/`center_id`. All user mutation is service-role. (The single most important thing — done right.)
- **Every server action re-checks `requireRole`/`requireUser`/`requireSuperAdmin` itself** — not protected by layout alone. Super-admin gate verifies `getUser().email` vs `SUPER_ADMIN_EMAIL` server-side, unforgeable, first line of every super-admin route/action.
- **Center-ownership re-checked before every service-role write** (students/classes/teachers/parents/worksheets/programs/lessons). Client never supplies `center_id`.
- **Parent/teacher read scoping correct** (parent → own child only; teacher → own classes) everywhere except the C1 admin branch.
- **Multi-step create rollbacks** present (createCenter/inviteTeacher/inviteParent/CSV row/createLesson/worksheet upload), rollback failures Sentry-captured. `updateLesson` upsert race fixed via unique `(lesson_id, student_id)`. Founding-slot race guarded by partial unique index.
- **Zod validation** applied on all risky write paths with length/range/enum/date caps. CSV parser handles BOM/quotes/escapes/blank rows. No destructured-and-ignored `{error}` on a critical write.
- **Secrets clean:** `.env.local`/`.vercel`/`.next` gitignored; service-role key only in `src/lib/supabase/admin.ts` (`import "server-only"`); no keys in source. **`tsc --noEmit`: 0 errors, `strict:true`, zero `any` in `src/`.** Git tree clean, nothing unpushed.
- **Sentry wiring complete** (client+server+edge, root + all 4 segment error boundaries + self-contained `global-error.tsx`, user-facing incident IDs). **i18n catalogs synced** (en/vi both 1333 keys, zero missing). robots/sitemap/manifest production-sane. **Not** vulnerable to CVE-2025-29927.
